# QuantDinger 回测内核 V2

本文说明当前回测系统的产品入口和工程边界。当前版本已经把“图表指标”和“可执行策略”拆开：指标只负责看图，脚本策略才负责回测、实盘和下单意图。

## 统一入口

- 人类前端入口：回测中心
- 回测创建接口：`POST /api/backtest/run`
- 回测历史接口：`GET /api/backtest/history`
- 回测详情接口：`GET /api/backtest/get`
- Agent/MCP 入口：`POST /api/agent/v1/backtest/run`

旧的 IDE 内嵌回测入口不再作为主要产品入口。指标 IDE 和策略 IDE 负责编辑、检查和保存资产；回测、历史记录、调参和结果分析集中在回测中心。

## 资产类型

### 图表指标

指标是 chart-only 资产，只输出图表内容：

- `output.plots`
- `output.signals`
- `output.layers`
- 可视化参数 `# @param`

指标不能直接回测，也不能直接发布成策略。指标代码中不要写订单、仓位、止盈止损、杠杆、回测区间、`# @strategy`、`df["open_long"]`、`df["close_long"]` 等执行内容。

如果用户想交易一个指标想法，必须先通过“AI 指标转策略”把它转换成 Script Strategy，然后在回测中心运行回测。

### 脚本策略

脚本策略是当前系统的标准可执行策略。它必须使用：

```python
def on_init(ctx):
    ...

def on_bar(ctx, bar):
    ...
```

策略通过显式订单意图表达交易行为：

- `ctx.open_long(...)`
- `ctx.close_long(...)`
- `ctx.open_short(...)`
- `ctx.close_short(...)`
- `ctx.add_long(...)`
- `ctx.add_short(...)`

两信号指标转成默认多头策略时，映射规则是：

```text
buy / golden cross  -> open_long
sell / death cross  -> close_long
```

不能把 `sell` 自动解释成反手做空。只有用户明确要求 short、both、reverse、hedge 等语义时，才生成空头相关意图。

### 模板策略

模板策略是脚本策略的预设。模板可以内置代码和默认参数，但运行时必须落到同一套 Script Strategy / Backtest Engine V2 链路，不能拥有独立回测分支。

## 撮合边界

当前系统使用 K 线数据回测，不使用历史逐笔成交和订单簿。因此回测必须遵守明确假设：

- 策略在当前 K 线收盘确认信号。
- 订单在下一根可处理 K 线开盘撮合。
- 同一根 K 线同时触发止盈和止损时，正式结果默认使用保守处理。
- 手续费、滑点、资金费率、杠杆和起止资金由回测中心配置统一处理。

策略代码不应该自行模拟资金账户，也不应该绕过回测引擎计算手续费、滑点、资金费率或成交账户。

## 现货与合约

现货规则：

- 固定 `leverage = 1`
- 固定只允许做多
- 不允许做空
- 资金费率固定为 `0`

合约规则：

- 支持做多、做空和双向
- 支持杠杆
- 支持手续费、滑点和资金费率
- 资金费率只在 `market_type=swap` 时生效

## 风控

风控分两层：

- 策略内退出信号：例如 `ctx.close_long(...)`、`ctx.close_short(...)`
- 回测引擎风控：止损、止盈、移动止盈、最大持仓 K 数等

百分比参数统一使用小数进入后端计算，例如 `0.03 = 3%`。前端如果显示为百分数，需要在进入引擎前转换。

## 发布约束

策略发布到市场前必须存在至少一条成功回测记录。没有成功回测记录的策略不能上架。

这个规则只适用于策略发布。指标发布仍然只按指标资产的可视化和质量检查规则处理，因为指标本身不参与回测和交易。

## 工程红线

- 不新增第二套回测引擎。
- 不让指标代码直接下单或直接回测。
- 不让模板策略拥有独立回测逻辑。
- 不用全 0 回测结果掩盖不可回测策略。
- 不允许策略代码自行维护回测资金账户。
- 不允许现货出现做空、杠杆或资金费率。
