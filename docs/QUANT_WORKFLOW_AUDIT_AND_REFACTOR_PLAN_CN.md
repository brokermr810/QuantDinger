# QuantDinger 策略、回测、实盘与交易机器人审查及改造计划

更新日期：2026-07-11

> 本文保留首轮审查证据和历史计划。当前完成状态、上线门槛和外部依赖以 `PRODUCTION_READINESS_CN.md` 为唯一基线。

## 1. 目标

把当前四块能力收敛成一条可验证、可晋级、可追溯的专业量化主链路：

```text
策略源码与版本
  -> 静态校验
  -> 可复现回测
  -> 模拟 / 仅信号观察
  -> 实盘前检查
  -> 实盘执行与状态监控
  -> 订单、成交、持仓和绩效复盘
```

专业工具的判断标准不是功能数量，而是同一策略版本在研究、回测、模拟和实盘中使用相同的数据口径、订单语义、风控口径和审计记录。

## 2. 本轮证据范围

本轮已完成：

- 前端路由、页面、API 调用和 i18n 结构盘点。
- 后端策略运行时、统一回测、执行器、实盘适配层及相关测试盘点。
- 前端生产构建验证。
- 四模块相关后端契约测试。
- 关键页面代码规模和 lint 基线检查。

暂未完成：

- 登录后的完整页面截图和逐步交互审查。最新开发环境触发了安全验证码，需要用户在浏览器中完成验证后继续。
- 真实交易所小额实盘验收。本轮不会在没有明确验收账户、限额和授权的情况下发出真实订单。

## 3. 总体判断

QuantDinger 已经有较完整的量化内核雏形，不需要推倒重来。后端的统一回测、ScriptStrategy、订单意图、执行队列、交易所适配、成交恢复和持仓同步方向是正确的。当前主要问题集中在四个方面：

1. 前端信息架构仍保留旧产品阶段的分裂入口。
2. 回测和实盘页面职责过重，主任务不够聚焦。
3. 交易机器人的创建、账户绑定、实例定位和验收链路需要闭环。
4. 后端虽然有较多契约测试，前端主链路几乎没有自动回归保护。

## 4. 四模块审查

### 4.1 策略页面

当前健康度：B

已有优势：

- 策略 IDE 已统一到 ScriptStrategy，支持源码、版本、校验、回测入口、实盘入口和发布。
- 页面目标相对清晰，目标文件约 75 KB，目标 lint 为 0。
- 已显式区分图表指标与可执行策略，避免把视觉 buy/sell 标记直接当成订单。
- 已具备代码校验、参数元数据、隐藏源码、版本恢复和指标转策略能力。

主要风险：

- “策略源码”“策略实例”“执行器实例”在用户认知上仍容易混淆。
- 缺少统一的生命周期状态：草稿、已校验、已回测、模拟观察中、可实盘、实盘中、异常。
- 策略版本与回测运行、实盘实例之间的绑定需要在 UI 中更明确，避免用户编辑源码后误以为运行中的实例已同步更新。

结论：策略 IDE 可以作为现有设计基线，不应重做。后续重点是补齐生命周期、版本绑定和晋级门槛。

### 4.2 回测页面

当前健康度：C+

已有优势：

- 后端已经统一到 BacktestEngine V2，并返回权益曲线、基准、订单、成交、闭合交易和执行假设。
- 页面已有交易回放、专业诊断、零成交提示、历史记录、参数覆盖、参数优化和样本外结果展示。
- 相关后端契约测试本轮通过。

主要风险：

- 单文件约 230 KB，模板、状态、数据适配、图表、历史、参数优化和样式都集中在一个页面。
- 目标文件仍有 27 个 lint error。
- 同一屏同时承担策略选择、市场配置、资金与费用、日期、代码参数、结果分析、交易回放、专业审查和参数优化，用户难以判断当前主任务。
- “运行一次可信回测”和“批量寻找最佳参数”是两个不同任务，目前并排放置，容易让首次使用者直接进入优化而忽略基线验证与过拟合风险。
- 没有前端数据适配测试，后端字段演进时容易出现静默降级或错误归一化。

改造方向：

- 保留左侧运行配置，但把主区拆为“结果”“诊断”“优化”“历史”四个明确工作区。
- 默认只展示一次回测结果；参数优化作为第二阶段任务，必须基于成功基线回测启动。
- 每次运行保存不可变输入快照：策略版本、标的、周期、日期、费用、滑点、杠杆、方向、参数和数据源。
- 把结果归一化、参数覆盖和交易配对提取为纯函数并补单元测试。

### 4.3 实盘页面

当前健康度：改造前 C，首批改造后 B-

已有优势：

- 后端已有策略列表、实时盈亏、持仓、订单、日志、复盘报告和账户快照能力。
- 新入口 `/strategy-center` 已建立，但此前主要是统计卡片加通用 Dashboard 嵌入。

主要风险：

- 真正的策略管理和详情仍由约 243 KB 的旧 `trading-assistant` 页面承担。
- 新实盘入口此前缺少可操作的运行实例表，用户必须从统计卡片跳回旧页面。
- 通用 Dashboard 更偏历史绩效，不等同于实盘运行监控。
- 缺少明确的运行健康字段：最后心跳、最后 K 线、最后信号、最后订单、最后错误、重启次数、对账状态和队列积压。
- 当前 `last_signal_time` 或 `updated_at` 不能替代运行时心跳，不能据此可靠判断卡死。

首批已落地：

- 在实盘首页增加当前用户的运行实例表、状态与执行模式筛选、搜索、资金与盈亏、最近活动时间和定时刷新。
- 新增隐藏的通用运行时详情路由 `/strategy-runtime`，避免脚本专用路由过滤掉执行器实例。
- 支持从实盘首页直接定位具体策略实例。

下一步：

- 新增专用 live operations API，返回真实运行健康、风险暴露、订单队列和对账状态。
- 把旧详情页拆成概览、持仓、订单与成交、绩效、日志和复盘几个独立组件。
- 增加账户级和策略级停止入口，但所有批量停止和实盘操作必须保留确认、审计和幂等保护。

### 4.4 交易机器人

当前健康度：改造前 C，首批改造后 B-

已有优势：

- 已支持 grid、DCA、martingale、layered martingale 四类交易机器人预览和实例生成。
- 后端执行器契约与路由测试覆盖了配置、脚本生成、创建和启动基础链路。
- 交易机器人是标准策略代码的配置、部署和运行实例，不是独立的第二套交易内核。
- 网格代码通过 `ctx.configure_robot(...)` 声明配置，宿主提供挂单、成交轮询和对账能力；DCA 与马丁代码直接使用标准策略上下文。

本轮发现的关键断点：

- 页面允许选择实盘模式，但此前没有保存账户选择，创建实例的 `exchange_config` 为空。
- 创建成功后跳转到脚本专用路由，而执行器实例的 `strategy_mode` 是 `bot`，因此会被过滤，用户看不到刚创建的实例。
- 自动预览请求可能乱序返回，旧配置结果有机会覆盖新配置结果。
- 前端没有执行器回归测试。

首批已落地：

- 实盘执行器必须绑定已保存的加密交易所账户。
- 后端拒绝没有 `credential_id` 的实盘执行器配置。
- 创建成功后进入通用运行时路由并定位新实例。
- 预览增加请求序号保护，避免旧响应覆盖新响应。
- 执行器页面目标 lint 从 23 个 error 降为 0。

下一步：

- 建立四类执行器的模拟验收矩阵：spot / swap、long / short、费用、部分成交、重启恢复、止盈、硬止损和断线重连。
- 对支持的交易所做离线契约测试，再选择测试网做真实 API 验收。
- 对网格单独验证挂单状态机；网格的 bar 回放结果不能冒充挂单撮合引擎的完全等价结果。

## 5. 横向架构问题

### 5.1 资产身份与版本

需要统一以下标识并贯穿所有记录：

```text
strategy_source_id
strategy_version_id
strategy_instance_id
backtest_run_id
strategy_run_id
order_intent_id
exchange_order_id
fill_id
```

用户必须能从任意实盘成交反查到策略版本和回测记录，也能从任意回测结果创建一个参数完全一致的模拟或实盘实例。

### 5.2 运行健康与可观测性

建议后端提供统一运行健康结构：

```text
runtime_status
last_heartbeat_at
last_bar_at
last_signal_at
last_intent_at
last_order_at
last_fill_at
last_error_code
last_error_at
restart_count
pending_intent_count
pending_order_count
reconciliation_status
```

页面应区分“正在运行”“等待下一根 K 线”“无新信号”“交易所连接异常”“订单积压”“持仓不一致”和“执行线程已停止”，不能都显示成一个 running 标签。

### 5.3 风控门槛

实盘前检查至少包含：

- 策略代码校验通过。
- 存在成功基线回测。
- 明确的费用、滑点和资金费率假设。
- 保存账户可用且市场、方向、杠杆兼容。
- 单策略资金上限、单订单上限、最大仓位、最大日亏损和最大回撤阈值。
- 同账户、同市场、同标的冲突检查。
- 仅信号或模拟观察期已完成，或用户明确覆盖该门槛。

### 5.4 前端工程质量

- 生产构建通过，但主要业务页包体较大。
- `backtest-center`、`executor-strategies`、`trading-assistant` 初始合计 60 个 lint error 和 331 个 warning。
- 前端原先没有单元测试脚本。
- 本轮已增加基于 Node Test Runner 的运行时归一化测试，后续应逐步覆盖回测 payload、结果归一化、执行器 payload 和生命周期判断。

## 8. 交付定义

当以下条件同时满足时，四模块才算形成专业量化闭环：

1. 用户能明确知道自己编辑的是源码、版本还是运行实例。
2. 每个实盘实例都能定位到唯一策略版本和成功回测。
3. 回测、模拟和实盘使用相同订单意图、方向、资金和风控语义。
4. 任一运行异常能在实盘中心被识别、解释和处理。
5. 任一订单和成交都有幂等键、状态轨迹和审计来源。
6. 核心流程有自动测试，并在每次构建中执行。
